Refresher on Branching, Cloning, Editing, Committing, Pushing, and Submitting a Pull Request on GitHub

DSHB Professional Development Conference — June 25, 2026

Author

Dr. Adam Ross Nelson

Session 4 · June 25, 2026 · 3:20 – 4:30 pm

Step One: Read, Re-read, and Read Again

Before you do anything else, read the starter instructions on this page in their entirety. Then read them again. Then a third time. The single most common failure mode for groups in this assignment is skimming the starter material once, getting stuck at minute 20, and then having to backtrack. Twenty minutes of careful reading at the start saves an hour of frustration later.

What You’re Teaching

As you may already know Git is the version-control system; GitHub is the collaboration platform built around it. You and your peers have probably touched both before, but for many this session is a chance to fill in the gaps that come from learning Git piecemeal. This session can help reinforce and solidify the previous lessons and exposure to Git and GitHub.

For this lesson you can refer back to previous lessons on Git and GitHub. Specifically The GitHub Confident Quizes Sandbox has helpful references. By the end of the session, your peers should be able to branch a repo, clone the branch, make changes, commit the changes, push the branch, and open a pull request against the main branch.

Mini Quick-Start

0. Make sure Git is installed, GitHub CLI is installed, and that gh auth login works

Sort this out before the session starts. Windows users recommended running in Git Bash so they match what macOS peers will run.

Check Git and gh installation:

git --version
gh --version

If Not Installed, Install Git and gh:

  • macOS (via Homebrew): brew install git gh
  • Windows (via winget): winget install --id Git.Git -e and winget install --id GitHub.cli

Verify and authenticate:

gh auth login    # choose GitHub.com → HTTPS → login with web browser
gh auth status   # should show a green check and your username

If gh auth status errors, fix it before moving on because nothing below will work without it.

1. Decide on Your Demo Repo

Don’t demo against a class repo with real history. Instead use a sandbox repo your group creates specifically for this purpose. It should have:

  • A simple README.md.
  • One or two text or code files peers can edit.
  • An open invitation for peers to contribute (e.g., “Add your name and a favorite tool to this list”).

Hint: Consider asking an angentic IDE to help you create your sandbox repository.

2. The Core Workflow

Your demo should walk through this complete loop:

First, on GitHub, fork the demo repo:

  1. Navigate to the demo repo in your browser.
  2. Click “Fork” (top right) to create a copy of the repository under your own GitHub account.

Then, in your terminal, clone your fork and make your changes:

# Clone your fork (replace yourusername with your GitHub username)
git clone https://github.com/yourusername/demo-repo.git
cd demo-repo

# Create a branch for your work
git checkout -b add-my-name

# Make your edit (open a file in any editor)
# ... edit ...

# Stage and commit your changes
git add favorite-tools.md
git commit -m "Add favorite tool from Maya"

# Push the branch to your fork
git push -u origin add-my-name

Finally, back on GitHub, open the pull request:

  1. Open your fork in the browser.
  2. Click “Compare & pull request.”
  3. Write a short PR description.
  4. Submit the pull request against the original repo.

3. The Conceptual Pieces Worth Pausing On

Most students can copy the commands. The conceptual pieces that actually matter:

  • What a branch is. A pointer to a commit, not a “copy of the code.”
  • The three-stage model. Working directory → staging area → commit. Show git status after each step.
  • What a pull request is for. It’s a request for review and discussion before code joins the main branch — not just a way to push code to GitHub.
  • What a merge conflict looks like. Demonstrate one on purpose and show how to resolve it.

4. Authentication

This is where most demos die. GitHub no longer accepts password authentication for git push. Your peers will need one of the following:

  • gh CLI configured with gh auth login (Recommended)
  • A personal access token (PAT), or
  • SSH keys configured.

Have them set this up before the hands-on portion. A 30-minute SSH-key detour kills the lesson.

  • The contributor never pushes directly to the main project repository; everything goes through their personal fork.
  • Pull requests are a review and discussion mechanism, not just a delivery mechanism — feedback loops back into more local edits.
  • Only the maintainer merges; the contributor’s job ends at “submit the pull request.”

Suggested 70-Minute Lesson Outline

  • 0 – 10 min — Intro. Why this tool matters for behavioral data science, what your peers will be able to do by the end of the session, and a 60-second preview of the demo.
  • 10 – 25 min — Live demo. You at the front, narrating clicks and commands. Show one complete success path from a clean state. Don’t try to cover edge cases yet.
  • 25 – 55 min — Hands-on. Peers do the demo themselves on their own machines. You and your group circulate to help. This is where most of the real learning happens — protect this time.
  • 55 – 65 min — Troubleshooting and gotchas. Surface the things that tripped you up during your own learning. What error messages should peers expect, and what do they actually mean? OS-specific differences? Auth problems?
  • 65 – 70 min — Wrap. Where to go next, one or two stretch challenges, pointers to the official documentation, and a Q&A buffer.
Rehearse the timing

Run through the lesson with your group at least once before the real session. Most first-time presenters underestimate the live-demo portion and overestimate how much hands-on time they actually have. Cut content rather than rush.

On Using the Official Documentation (Not GenAI) as Your Primary Tutor

For this assignment, treat the tool creator’s official documentation as your primary teacher. There is a strong temptation to ask Claude, ChatGPT, or another generative AI to “just explain it to me.”

Relying on GenAI in this way during the earliest stages of adopting a new tool is a mistake. Learn from the documentation first, and then use GenAI to help you understand specific parts of the documentation or to clarify confusing concepts.

Here is why:

  • Docs are the authoritative source. GenAI tools paraphrase, summarize, and occasionally hallucinate. The official documentation is what the tool’s authors actually wrote about how the tool actually behaves.
  • Reading docs is the skill we’re teaching. The whole premise of this conference is that successful long-term scientists are the ones who can pick up a new tool from its documentation. If you outsource that to an AI, you miss the practice rep.
  • You’ll be a better teacher. When you’ve read the source material carefully, you can answer the questions your peers ask in the hands-on portion. When you’ve only skimmed an AI summary, you can’t.

Suggestion: When getting started out, use GenAI as a sidekick (to debug a specific error, to explain a single unfamiliar term). Do not use it as your tutor of first resort.

Topic-Specific Demo Tips

  • As you set Git and GitHub up for yourselves, keep track of common mistakes, errors, misunderstandings, and points of friction. Designate a member of your team to centrally collect authentication errors, merge-conflict messages, “I’m stuck in vim” moments, and any Mac vs. Windows differences. Then write them up into a one-page “troubleshooting guide” you can share with the others in class on the day of via the course chat platforms.
  • Use a “team teaching” approach. To make this lesson successful, every single member of your team will need to be confident running gh auth login, cloning a repo, branching, committing, pushing, opening a pull request, and resolving a basic merge conflict. The members of your team will need to circulate through the room helping other students. There will be many stumbling blocks.
  • Audit auth setup at the start. Spend the first 5 minutes of hands-on time checking that everyone can git push a trivial change. If three people can’t, the rest of the session is downhill.
  • Lean into the code-review angle. This session naturally folds in reading other people’s code and code review — when you walk through opening a PR, talk about how to review one. What does a useful review comment look like?
  • Show the GitHub web UI alongside the command line. Some peers will prefer the GUI; others will prefer the CLI. Both are valid; both will keep showing up in their careers.
